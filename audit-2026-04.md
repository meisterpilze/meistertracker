## Phase 1 — Security

11 findings: 2 HIGH, 6 MED, 3 LOW. No live RCE, no plaintext passwords on disk, no session-cookie disasters. Full draft in `phase-1-security.md`.

### S-01 — MCP transport is role-blind; any token = full admin [HIGH] `OPEN`
`mcp-server.js` (whole file — no role gates) + `server.js:124-141` (`checkMcpAuth` returns boolean, drops the OAuth token's `userId`) + `server.js:3810-3815` (OAuth `/authorize` GET allows any logged-in user, not just admins).

The MCP tool surface exposes destructive operations (`delete_batch`, `delete_culture`, `delete_mushroom_strain`, `manage_zones { delete }`, `manage_racks`, `update_inventory`, `assign_barcode`, etc.) and never inspects who the caller is. Two paths reach the transport:
1. **Legacy static token** (`db.getMcpToken`) — single global API token, anyone holding it gets full admin.
2. **OAuth access token** — `/oauth/authorize` requires only a session cookie, so a worker can grant a Claude client an access token, which then enjoys the same admin-level MCP surface.

**Impact**: A worker who installs Claude Desktop and clicks "Allow" escalates to full DB-mutation rights. The web UI gates the same operations behind `requireAdmin` (e.g. `server.js:4721, 5340, 5448, 5538, 5611`).

**Fix sketch**: (a) `checkMcpAuth` returns `{userId, role}` instead of `true/false`; (b) wrap each MCP tool with a required-role check (or a wrapper around `server.tool`); (c) restrict `/oauth/authorize` to admins, or scope OAuth-derived tokens to a read-only tool subset.

### S-02 — `/api/webhook/github` crashes server on bad-length signature [HIGH] `OPEN`
`server.js:5921` inside an unguarded `req.on('end', () => { ... })`.

`crypto.timingSafeEqual(Buffer.from(sig), Buffer.from(expected))` throws `RangeError` if the attacker-controlled `X-Hub-Signature-256` header isn't 71 chars. The throw escapes the callback → `uncaughtException` handler at `server.js:7083` → `process.exit(1)`.

**Impact**: One unauthenticated POST per ~PM2-restart-window keeps the server in a permanent crash loop. Only active when `GITHUB_WEBHOOK_SECRET` is configured (the auto-deploy flow added in #340).

**Fix sketch**:
```js
const sigBuf = Buffer.from(sig);
const expBuf = Buffer.from(expected);
if (sigBuf.length !== expBuf.length || !crypto.timingSafeEqual(sigBuf, expBuf)) {
  jsonErr(res, 401, 'bad signature'); return;
}
```
Also wrap the `req.on('end')` body in `try/catch`.

### S-03 — Worker-writeable endpoints that should be admin [MED] `OPEN`
- `server.js:5111` POST `/api/cultures`
- `server.js:5140` POST `/api/cultures/:id/rename`
- `server.js:5167` PATCH `/api/cultures/:id`
- `server.js:5184` DELETE `/api/cultures/:id` — orphans `batches.source_id` (no FK cascade in `db.js:2457`)
- `server.js:5386` POST `/api/assets` — workers can upsert assets up to `purchasePrice: 100,000,000`
- `server.js:4799` DELETE `/api/scan-log/:id` — comment says "by design" but lets workers delete *anyone's* scan entries

**Impact**: A disgruntled worker can wipe lab cultures or fabricate asset purchases. No actor in audit log.

**Fix sketch**: `if (requireAdmin(req, res)) return;` on each. For scan-log delete, mirror the contamination-report ACL at `server.js:5044-5048` (owner-or-admin).

### S-04 — Backup encryption: weak password floor + scrypt cost [MED] `OPEN`
`server.js:6326, 6342, 6406` — `scrypt` with `N: 32768, r: 8, p: 1` and 8-char password minimum; redundant outer HMAC-SHA256 over an already-AEAD ciphertext using the same key (`server.js:6348`).

**Impact**: An attacker who exfiltrates an `.enc` backup and an 8-char password dictionary can decrypt the entire DB (sessions, OAuth tokens, DuckDNS material).

**Fix sketch**: `N: 131072, r: 8, p: 4` (~1s on a modern server); enforce ≥ 12-char password with complexity check; drop the redundant HMAC.

### S-05 — Session cookies missing `__Host-` prefix; no rotation on login [MED] `OPEN`
`server.js:458-468, 4127`. Cookies are `HttpOnly; SameSite=Strict; Path=/; Secure (https)` — solid baseline. Two refinements:
1. No `__Host-` prefix → a sibling subdomain on the same eTLD+1 could plant a cookie.
2. `/api/auth/login` issues a new token but doesn't `db.deleteSession` any pre-existing token presented in the request.

**Fix sketch**: Switch cookie name to `__Host-session`; on login, delete the old token before minting the new one.

### S-06 — `setupRequired` race lets attacker claim admin [MED] `OPEN`
`server.js:4065-4096`. First-run gate is only `db.countUsers === 0`. On a fresh public deployment, an unauthenticated remote attacker who hits `/api/auth/setup` first becomes admin.

**Fix sketch**: Print a setup token to stdout on first start (require it as a header), or bind setup to `127.0.0.1`/`::1` (mirror the pattern at `server.js:4400-4411`).

### S-07 — Print-bridge token compared with `-eq` (not constant time) [MED] `OPEN`
`scripts/print-bridge.ps1:419-421`. Also: bridge accepts unauthenticated `POST /print` if `$Token` is empty (warning is console-only at line 440).

**Fix sketch**: Use `[System.Security.Cryptography.CryptographicOperations]::FixedTimeEquals` (PS 7+) or manual XOR-and-fold; refuse to start if `$Token` is empty AND the bind isn't `127.0.0.1`.

### S-08 — Static MCP token: no rotation history, no last-used, ambiguous encoding [MED] `OPEN`
`db.js:3173-3186` + `server.js:132`. `generateMcpToken` overwrites the prior hash on every call; no `last_used`, `created`, `revoked` columns; `Buffer.from(hexString)` is invoked without specifying `'hex'` encoding, so it works only because both inputs are equal-length UTF-8 strings.

**Fix sketch**: Treat the static token as deprecated in favour of OAuth; add audit columns; pass `'hex'` to `Buffer.from` for clarity.

### S-09 — Path-traversal sanitiser is platform-dependent [LOW] `OPEN`
`server.js:1479-1483`. `sanitizePart` relies on `path.basename`, which on POSIX treats `\` as a literal. No exploit today; brittle invariant.

**Fix sketch**: Reject any segment containing `/`, `\`, `\0`, or starting with `.` explicitly.

### S-10 — `gen-cert.sh` interpolates `$DOMAIN` into OpenSSL config unescaped [LOW] `OPEN`
`gen-cert.sh:34-66`. Local-admin-only attack surface today; would matter if `CERT_DOMAIN` is sourced from untrusted env later.

**Fix sketch**: Whitelist `^[A-Za-z0-9.-]+$` before interpolation.

### S-11 — `/api/health` over-shares to authenticated workers + Host-header reflection [LOW] `OPEN`
`server.js:4354-4363, 284`. Verbose health payload (Node version, platform, memory, SSE clients, backup details) goes to *any* authenticated user. `req.headers.host` is reflected into OAuth issuer metadata.

**Fix sketch**: Restrict verbose health to admins; validate `Host` against a configured `PUBLIC_HOSTNAME` env var.

## Phase 2 — Data integrity & domain correctness

23 findings: **9 HIGH, 9 MED, 5 LOW**. The **inventory ledger clamp is NOT fixed server-side** despite `pain-points.md` flagging it (only the client-side pre-flight in #371 shipped). **`/api/caldav/sync` is an active corruption vector** — every sync silently wipes ~6 columns. Full draft in `phase-2-integrity.md`.

### I-01 — `applyInventoryDelta` records unclamped delta while running stock is clamped to 0 [HIGH] `OPEN`
`db.js:2752-2774`. `UPDATE inventory SET ${col} = MAX(0, ${col} + ?)` clamps the running balance, but the immediately-following `INSERT INTO inventory_log` writes the bare `deltaKg`. Stock 50, request -200 → running becomes 0, log records -200. `sum(deltas) ≠ stock`. `deleteBatchById` (`db.js:2202-2238`) reverses the same unclamped deltas, **crediting phantom stock** when an over-committed batch is deleted. Test at `test/db.test.js:504-507` codifies the broken behaviour (must be updated as part of the fix).

**Fix sketch**: clamp `deltaKg` to `max(deltaKg, -currentStock)` for negative deltas, record clamped value, surface the gap in the API response.

### I-02 — Batch creation spans 2-N HTTP calls with no client-side rollback [HIGH] `OPEN`
`app.js:7058-7110, 3789-3791`. `POST /api/batches` then a loop of per-material `POST /api/inventory/delta`. Network drop after step 1 = ghost batch with no inventory deduction. Step-2 partial failure (hardwood applied, wheatbran fails) leaves no rollback; client optimistically mutated `inventory.stock` so the UI looks fine while the server didn't write.

**Fix sketch**: extend `db.insertBatch` to take a `deltas[]` array, apply inside the same SQLite transaction, drop the separate `/api/inventory/delta` calls from the create path.

### I-03 — Contamination report runs 4 unrelated writes outside a transaction [HIGH] `OPEN`
`server.js:5017-5067`, `db.js:4178-4193`. `createContaminationReport` INSERT → `addContaminationPhoto` INSERTs → `appendScanEntries(MOVE)` (its own internal txn) → `setContaminationReportScanLogId` UPDATE. Failure between steps yields: orphaned report, MOVE in `scan_log` linked to nothing, bag in CONTAM zone with no audit link back.

**Fix sketch**: wrap the DB writes in `db.exec('BEGIN')`/`COMMIT` with `ROLLBACK` in catch.

### I-04 — `writeAll` silently drops 6+ schema columns; **CalDAV sync corrupts data on every cycle** [HIGH] `OPEN`
`db.js:1426-1828` (`writeAll`), `server.js:6805` (`/api/caldav/sync` calls `writeData(data)`), `server.js:4678` (`/api/data POST`).

The full-state UPSERT path's column lists haven't been kept in sync with schema migrations:
- `scan_log` insert at `db.js:1502` omits `reason` (added v24)
- `harvests` insert at `db.js:1523` omits `quality`, `notes` (added v26)
- `batches` upsert at `db.js:1446` omits `strain_id` (v19), `strain_text` (v21)
- `cultures` upsert at `db.js:1547` omits `strain_id` (v19), `strain_text` (v30)
- `calendar_events` upsert at `db.js:1715` omits `exception_dates` (v32)

`/api/data POST` is admin-only, but `/api/caldav/sync` is reachable by any authenticated user. **Every CalDAV sync round-trip wipes contam reasons, harvest notes/quality, strain linkages, and recurring-event exceptions.** This is happening in production now if anyone uses CalDAV.

**Fix sketch**: add the missing columns to each UPSERT and the SELECT in `readAll`. Long-term: generate the column list from a single source-of-truth list.

### I-05 — `traceLineageBack` has no cycle protection — infinite loop on `parent_id` cycle [HIGH] `OPEN`
`db.js:3957-3993`, invoked from `mcp-server.js:1837` (`trace_lineage`). The culture branch follows `parent_id` in a `while (current)` loop with no `visited` Set. Schema (`db.js:61-71`) doesn't FK-link `parent_id`, and `POST /api/cultures` (`server.js:5269-5295`) doesn't validate parent. A self-cycle or mutual cycle blocks the Node event loop until process kill. The forward variant `traceLineageForward` (`db.js:3996-4046`) is correctly cycle-safe.

**Fix sketch**: add `visited` Set; reject `parent_id == id` and existing-cycle inputs in `insertCultures`/`updateCulture`.

### I-06 — `renameBatch` doesn't update `contamination_reports` [HIGH] `OPEN`
`db.js:2123-2153`. Renames batches/bags/scan_log/harvests/inventory_log.ref/barcodes. Misses `contamination_reports.batch_id` and `.bag_id` (no FK). Filter-by-batch in the contam list (`db.js:4222-4225`) silently misses renamed batches.

**Fix sketch**: add `UPDATE contamination_reports SET batch_id=? WHERE batch_id=?` and the `bag_id REPLACE` analogue.

### I-07 — `deleteBatchById` doesn't clean up `contamination_reports` [HIGH] `OPEN`
`db.js:2202-2238`. After delete, contam reports with `batch_id = '<deleted>'` linger forever. UI shows `(unknown batch)`. If the batch ID is recycled, old reports re-attach.

**Fix sketch**: add FK with `ON DELETE SET NULL` (cleanest), or explicit DELETE in `deleteBatchById`.

### I-08 — `/oauth/token` exchange not atomic [MED] `OPEN`
`server.js:3816-3837, 3856-3877`. `markOAuthCodeUsed` then 2× `createOAuthToken` outside a transaction. Partial failure leaves auth code burned with broken refresh chain.

**Fix sketch**: wrap in BEGIN/COMMIT with ROLLBACK in catch.

### I-09 — KPI snapshot day window is UTC-naive; near-midnight events bucket wrong [MED] `OPEN`
`db.js:3699-3710`. `today = new Date().toISOString().slice(0, 10)` is UTC; lab is in Berlin (UTC+1/+2). Batches created 22:30-23:59 UTC on day N (which is 00:30-01:59 Berlin local on day N+1) bucket into UTC-day N. Same for the production-pipeline `due` comparison (out-of-scope obs).

**Fix sketch**: compute "today" in lab-local time via `getTimezoneOffset`, document the chosen day boundary.

### I-10 — `getStatus` MOVE handler is asymmetric → out-of-order replay produces phantom +1 [MED] `OPEN`
`app.js:4500-4515`. `from`-side decrement is `Math.max(0, ...)`-clamped; `to`-side increment is unconditional. With out-of-order offline replay (MOVE arrives before ADD), the to-zone increments twice net.

**Fix sketch**: gate every increment on `lastScanByBag` (already a map, just consult it). Or fold replay into a single "set bagZone[bag]=current_zone" pass.

### I-11 — Offline scan-log queue has no idempotency key; retries produce duplicates [MED] `OPEN`
`sw.js:94-118`, `db.js:2289-2318`. Server crash mid-write → SW sees network error → keeps queued → next replay creates duplicate row.

**Fix sketch**: client-generated `client_uuid` per scan, UNIQUE INDEX, `INSERT OR IGNORE` server-side.

### I-12 — Out-of-order replay overwrites legitimate admin edits with stale offline scans [MED] `OPEN`
`sw.js:94-118`, `server.js:4906-4923`. Worker queues MOVE INC→FRUIT1 offline. Admin moves bag to CONTAM via web UI. Worker reconnects, queue replays, **bag returns to FRUIT1**, undoing the contamination quarantine.

**Fix sketch**: include `expected_current_zone` in queued POST; server returns 409 on mismatch; client surfaces a "rejected" notification.

### I-13 — `contam_rate_pct` can exceed 100% [MED] `OPEN`
`db.js:3737-3758`, `app.js:4682-4689`. Numerator counts bags whose `to` matches a contam zone; denominator counts bags with an `ADD` event. A bag with no ADD (lost via I-04) but a MOVE-to-contam appears in numerator only.

**Fix sketch**: scope numerator to bags that also appear in the ADD set.

### I-14 — `getProductionPipeline` REMOVE handler diverges from `getStatus` [MED] `OPEN`
`db.js:4076-4083` (server: unconditional `delete bagZone[e.bag]`) vs `app.js:4514` (client: only on `e.from && c[fz]`). Dashboards disagree on bags with NULL `from` REMOVE entries.

**Fix sketch**: pick server's "REMOVE = bag gone" semantics; drop client `e.from` guard.

### I-15 — VEVENT generator emits no SEQUENCE, RRULE, or EXDATE [MED] `OPEN`
`server.js:1815-1849, 1862-?, 1952-2010`. None of `taskToVTODO`, `customEventToVEVENT`, `batchToVEVENT` emit SEQUENCE (RFC 5545 update detection), RRULE (recurrence definitions are stored on tasks/events but not emitted), or EXDATE (per-occurrence exceptions stored in v32 column never propagate). External CalDAV clients see only single occurrences; deletes of single occurrences come back on next sync.

**Fix sketch**: add `sequence` column + emit; translate `recurrence` enum to `RRULE:FREQ=...` with `UNTIL=` from `recurrence_until`; emit `EXDATE;TZID=Europe/Berlin:` from `exception_dates`.

### I-16 — `oauth_codes`/`oauth_tokens.user_id` have no FK; user delete leaves live tokens [MED] `OPEN`
`db.js:408-428, 2016-2019`. `deleteUser` removes session rows + user row but not OAuth tokens. After delete, the user retains MCP access until token expiry (1h access / 30d refresh). Combined with Phase 1 S-01 this is a security-flavoured integrity bug.

**Fix sketch**: add `ON DELETE CASCADE` via migration; `deleteUser` wrapped in BEGIN/COMMIT.

### I-17 — Backup restore: no mutex, closed-handle window, racy concurrent restores [MED] `OPEN`
`server.js:6692-6713`. Between `database.close()` and reassign, concurrent requests throw closed-handle errors. No serialization on multiple admin restore POSTs.

**Fix sketch**: module-scoped `restoreInProgress` flag; reject 503 on second attempt; queue or fail incoming requests during the window.

### I-18 — `_printViaPowerShell` tempfile collides on simultaneous prints [MED] `OPEN`
`server.js:1405-1409`. `Date.now()` filename → two prints in same ms collide. Already noted as Phase 1 forwarded item.

**Fix sketch**: `crypto.randomBytes(8).toString('hex')` (mirrors backup-restore tempname).

### I-19 — Substrate composition `< 100%` silently accepted, gap unaccounted [LOW] `FIXED (Phase 2 LOW PR)`
`app.js:7004-7012`, `db.js:insertBatch`. Client now alerts on `Math.abs(hw + wb - 100) > 0.01` (both directions) instead of only `> 100`. Server `insertBatch` mirrors the check for block batches via API/MCP. Skipped when both percentages are zero (legacy / no composition tracked) or `batchType !== 'block'` (grain spawn).

### I-20 — `cultures.parent_id` accepts self-reference and type mismatches [LOW] `FIXED (Phase 2 LOW PR)`
`db.js:VALID_CULTURE_PARENT_TYPES`, `validateCultureParent`. New helper in `db.js` codifies the lineage rules (MC→none, PD→MC|PD, LC→MC|PD|LC, G2G/GS→MC|PD|LC|G2G|GS) and is called from `insertCultures` and `updateCulture`. Self-cycle reject from the HIGH PR remains in place. Existing rows are not retroactively cleaned up.

### I-21 — `getStatus` REMOVE ignores entries with NULL `from` [LOW] `FIXED (closed by I-10)`
`app.js:4500-4524`. The I-10 rewrite (MED PR) replaced the per-event delta logic with a `lastByBag` map that derives counts from the most recent scan per bag. The old `e.from`-guarded REMOVE branch no longer exists, so the NULL-`from` edge case is moot.

### I-22 — `inventory_log` has no `user_id`; "who changed inventory" is unknowable [LOW] `FIXED (Phase 2 LOW PR)`
`db.js:inventory_log` schema, migration v40. `user_id INTEGER REFERENCES users(id) ON DELETE SET NULL` added. `applyInventoryDelta`, `applyInventoryDeltaNoTxn`, `setInventoryAbsolute`, `insertBatch`, `addBagsToBatch`, `deleteBatchById` all accept a `userId` param and forward to the INSERT. Server (`/api/inventory/delta`, `/api/inventory/set`, `POST /api/batches`, `PATCH /api/batches/:id/bags`, `DELETE /api/batches/:id`) and MCP (`update_inventory`, `create_batch`, `add_bags_to_batch`, `delete_batch`) thread `auth.userId` through. `readAll` and `getInventory` surface `user_id` in the inventory log payload.

### I-23 — `addBagsToBatch` does NOT touch inventory; growing a batch bypasses the ledger [LOW] `FIXED (Phase 2 LOW PR)`
`db.js:addBagsToBatch`, `db.js:computeBatchMaterialDeltasForKg` (new helper). Reads the existing batch row, computes proportional deltas for the new bags' wet weight (reusing the batch's stored substrate composition), and applies them via `applyInventoryDeltaNoTxn` inside the existing BEGIN/COMMIT. The `updateBatchField` comment about why `qty` is excluded was made accurate (it referenced ledger entries that didn't yet exist).

## Phase 3 — Reliability & ops

23 findings: **6 HIGH, 12 MED, 5 LOW**. Restore drill **FAILED**. Two production-active issues:
- **17 consecutive days of zero successful auto-backups** (auto-backup deletes its own output during rotation)
- **Every historical `.enc` backup is unrecoverable** (SCHEMA creates a partial index against a column that migration v39 hasn't added yet)

The lab has been protected only by OneDrive accidentally syncing the manual `update_server.sh`-created `meistertracker_*.db` files, which use `cp` (not VACUUM INTO) and aren't WAL-consistent. Full draft in `phase-3-reliability.md`.

### R-01 — Auto-backup rotation deletes the just-written file [HIGH] `OPEN`
`server.js:658-686`. `runDailyBackup` writes `meisterpilze_backup_<date>.db`, then rotation `fs.readdirSync(...).filter(.endsWith('.db')).sort()` slices alphabetically — `meisterpilze_b...` sorts BEFORE the 30 pre-existing `meistertracker_*.db` manual backups, so rotation deletes today's auto-backup. `verifyBackupFile` then errors with `stat ENOENT`. `.backup-status.json` records this every day since 2026-04-12.

**Fix**: rotation filter to `^meisterpilze_backup_\d{4}-\d{2}-\d{2}\.db$` only. Run rotation BEFORE the new write. Refuse to delete files written < 60s ago.

### R-02 — `verify-backup.js` and `/api/backup/restore` cannot open ANY pre-v39 backup [HIGH] `OPEN`
`db.js:60-61` SCHEMA, `db.js:967-983` migration v39, `db.js:1108-1126` openDb. SCHEMA contains `CREATE UNIQUE INDEX ... ON scan_log(client_uuid) WHERE client_uuid IS NOT NULL`. Run before `runMigrations`. On a pre-v39 DB the column doesn't exist → `no such column: client_uuid`. Verified against the live prod DB and 3 backup files. `POST /api/backup/restore` runs the same `openDb` validation — encrypted-backup restore via the admin UI fails for any historical archive.

**Fix**: move the partial-index creation OUT of SCHEMA into migration v39. Add `openDb({ skipMigrations: true })` mode for the verify script.

### R-03 — `verify-backup.js` only tests fresh DBs, not real backups [HIGH] `OPEN`
`scripts/verify-backup.js:38-71`. Script (1) opens prod DB, (2) `db.backupDb` to scratch, (3) re-opens scratch. Never opens a real `backups/*.db` file as a restore source. Returns exit 0 whenever the live DB opens — even when 30 backup files in the dir are unrestorable.

**Fix**: add `--restore-test <backup-file>` mode that copies a real backup, opens via the same `openDb` path used by `/api/backup/restore`, reports row counts.

### R-04 — `check-backup-health.js` looks for the wrong filename pattern [HIGH] `OPEN`
`scripts/check-backup-health.js:80`. Filter is `f.startsWith('meisterpilze_backup_')`. But manual backups (`update_server.sh backup_data`, `START.bat`, `pre_collapse_*`) use `meistertracker_*.db`. Health check sees zero "valid" backups when 30 manual ones exist.

**Fix**: accept BOTH prefixes. Print a per-prefix age table.

### R-05 — DEPLOYMENT.md has no restore section [HIGH] `OPEN`
`DEPLOYMENT.md:320-340`. Section 9 documents health/verify scripts but no restore walk-through. New operator following only DEPLOYMENT.md cannot find the restore path. README.md has it but DEPLOYMENT.md doesn't link to it.

**Fix**: add a "Restoring from a backup" section with three subsections (admin UI .enc restore, manual file swap, WAL-only recovery). Cross-link from README.

### R-06 — No off-site backup automation; OneDrive happens by accident [HIGH] `OPEN`
`DEPLOYMENT.md:333-340`, `README.md:141`. README says OneDrive auto-syncs `backups/`, but that's a side-effect of Julian's working directory. A Linux deploy has zero off-site copy unless the operator manually adds the documented cron line.

**Fix**: pick ONE official off-site path, document as REQUIRED, expose `/api/health.backup.offSite.lastSync`.

### R-07 — `unhandledRejection` only warns, `uncaughtException` exits — asymmetric [MED] `OPEN`
`server.js:7558-7568`. Throw inside Promise without `.catch()` warns + continues; same throw outside → exit. With Phase 1 S-02 in place (defensive try/catch), the actual exit is now mostly defensive — recommend warn-only for both. **Fix**: align policies, or `STRICT_PROMISES=1` env to escalate.

### R-08 — MCP/OAuth rate limiter ignores TRUST_PROXY [MED] `OPEN`
